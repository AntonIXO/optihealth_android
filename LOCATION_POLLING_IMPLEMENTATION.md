# Low-Power Location Polling Implementation

## Overview

This document describes the implementation of a battery-efficient background location polling service for PIHS. The service is designed to periodically log the user's approximate location with minimal to zero perceptible battery drain. This data is the foundation for the server-side "Anchor" engine, which automatically discovers significant locations like "Home" and "Work" without any user friction.

## Architecture

### Key Components

1. **LocationPollingWorker** (`location/LocationPollingWorker.kt`)
   - WorkManager worker that executes periodic location polls
   - Uses `PRIORITY_BALANCED_POWER_ACCURACY` for low-power location requests
   - Polls approximately every 30 minutes (inexact interval)
   - Uploads location data to backend via DataUploaderService

2. **LocationManager** (`location/LocationManager.kt`)
   - Manages location polling lifecycle
   - Handles WorkManager work scheduling and cancellation
   - Manages location permissions
   - Uses `ExistingPeriodicWorkPolicy.KEEP` to prevent duplicate work

3. **LocationBootReceiver** (`location/LocationBootReceiver.kt`)
   - BroadcastReceiver for `ACTION_BOOT_COMPLETED`
   - Re-enqueues location polling work after device reboot
   - Checks permissions before re-starting

4. **PIHSApplication** (`PIHSApplication.kt`)
   - Initializes location polling on app start if previously enabled
   - Ensures WorkManager work persists across app restarts

## Battery Efficiency Features

### WorkManager Constraints

The location polling work is configured with the following constraints:

```kotlin
val constraints = Constraints.Builder()
    .setRequiredNetworkType(NetworkType.CONNECTED) // Only run if device has internet
    .setRequiresBatteryNotLow(true) // Do not run if battery is dying
    .build()
```

### Low-Power Location Priority

The worker uses `Priority.PRIORITY_BALANCED_POWER_ACCURACY`:
- Uses Wi-Fi and cell tower triangulation (no GPS)
- Accuracy: ~100m-300m (sufficient for location clustering)
- Minimal battery impact

### Inexact Interval

The 30-minute polling interval is inexact, allowing Android to batch location requests with other apps for maximum efficiency.

## Permission Flow

### Required Permissions

1. **ACCESS_COARSE_LOCATION**: Primary permission for low-power location access
2. **ACCESS_BACKGROUND_LOCATION**: Required for location polling when app is not in use
3. **RECEIVE_BOOT_COMPLETED**: For persistence across reboots

### User-Facing Permission Requests

The UI guides users through a two-step permission flow:

1. First: Request `ACCESS_COARSE_LOCATION` (foreground location)
2. Second: Request `ACCESS_BACKGROUND_LOCATION` with rationale:
   - "To automatically find your 'Home' and 'Work' to give you better insights"

## Data Upload

### Location Data Structure

Location data is uploaded as a `DataPoint` with the following structure:

```json
{
  "metric_source_id": <auto-generated>,
  "timestamp": "2025-10-26T18:11:00Z",
  "metric_name": "environment_location",
  "value_geography": {
    "type": "Point",
    "coordinates": [longitude, latitude]
  }
}
```

### Metric Source

A unique metric source is created per user with:
- `source_identifier`: "android_location_polling_v1"
- `source_name`: "Android Location Polling"

The worker automatically gets or creates this metric source on first run.

### Backend Integration

Location data is uploaded via the existing `DataUploaderService`, which POSTs to the Supabase Edge Function endpoint for data ingestion.

## Error Handling

The worker implements proper error handling with retry logic:

- **Result.success()**: Location fetched and uploaded successfully
- **Result.failure()**: Non-retryable error (e.g., user not authenticated, missing permissions)
- **Result.retry()**: Retryable error (e.g., network failure, location timeout)
  - Maximum 3 retry attempts with exponential backoff

## Removed Components

The following components were removed as part of this refactor:

1. **LocationTrackingService** - Foreground service with persistent notification
2. **Battery optimization exemption** - No longer needed with WorkManager
3. **Foreground service permissions** - Not required for background work
4. **Notification permissions** - No persistent notification required

## Configuration

### Polling Interval

To change the polling interval, modify `POLLING_INTERVAL_MINUTES` in `LocationManager.kt`:

```kotlin
companion object {
    private const val POLLING_INTERVAL_MINUTES = 30L // Default: 30 minutes
}
```

Note: WorkManager enforces a minimum interval of 15 minutes for periodic work.

### Location Priority

To change location priority (not recommended), modify the priority in `LocationPollingWorker.getCurrentLocation()`:

```kotlin
fusedLocationClient.getCurrentLocation(
    Priority.PRIORITY_BALANCED_POWER_ACCURACY, // Change priority here
    null
).await()
```

Available priorities (ordered by power consumption):
- `PRIORITY_PASSIVE` - Lowest power, may take very long
- `PRIORITY_LOW_POWER` - City-level accuracy (~10km)
- `PRIORITY_BALANCED_POWER_ACCURACY` - Block-level accuracy (~100m) **[Current]**
- `PRIORITY_HIGH_ACCURACY` - Highest accuracy, uses GPS **[DO NOT USE]**

## Backend Requirements

### Database Schema

Ensure the following table structure exists:

**metric_sources**
- `id` (bigint, primary key)
- `user_id` (uuid, foreign key)
- `source_identifier` (text, unique per user)
- `source_name` (text)
- `is_active` (boolean)
- `last_synced_at` (timestamp)

**data_points**
- `id` (bigint, primary key)
- `metric_source_id` (bigint, foreign key)
- `timestamp` (timestamptz)
- `metric_name` (text)
- `value_geography` (geography(Point,4326))
- Additional fields: value_numeric, value_text, value_json, unit, tags

### Edge Function

The existing data ingestion Edge Function should handle location data with the `value_geography` field in GeoJSON format.

## Troubleshooting

### Location data not appearing in backend

1. Check worker status: `adb shell dumpsys jobscheduler | grep LocationPolling`
2. Check logs: `adb logcat | grep LocationPollingWorker`
3. Verify permissions are granted
4. Ensure device has network connectivity
5. Verify battery is not low

### Work not running after reboot

1. Check BootReceiver logs: `adb logcat | grep LocationBootReceiver`
2. Verify `RECEIVE_BOOT_COMPLETED` permission in manifest
3. Check if tracking was enabled before reboot

### High battery usage

1. Verify `PRIORITY_BALANCED_POWER_ACCURACY` is being used
2. Check polling interval is at least 30 minutes
3. Ensure WorkManager constraints are properly set
4. Review logs for excessive retries or errors
