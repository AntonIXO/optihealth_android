PIHS: Supabase Database Schema & Data Logging SpecificationThis document details the proposed PostgreSQL database schema for the Personalized Insight Health Server (PIHS) project, hosted on Supabase. It also outlines how various types of health data should be logged into these tables.1. Core PrinciplesUser-Centric Data: All data is tied to a specific authenticated user. Supabase's Row Level Security (RLS) will be heavily utilized to ensure users can only access and modify their own data.Time-Series Focus: The data_points table is central for storing most health metrics over time.Flexibility: The schema uses TEXT for metric_name and JSONB for value_json and tags in data_points to accommodate a wide variety of current and future health data types.Standardization: Clear conventions for metric_name are crucial for consistent querying and analysis.Relational Integrity: Foreign keys will be used to maintain relationships (e.g., between data_points and metric_sources).2. Table DefinitionsAll tables will reside in the public schema of your Supabase PostgreSQL database unless specified otherwise. The auth.users table is provided by Supabase Auth.2.1. users (Provided by Supabase Auth)This table is automatically managed by Supabase Authentication.Key Columns:id (UUID, Primary Key): Unique identifier for the user. This will be the primary foreign key in other tables to associate data with a user.email (TEXT): User's email.Other Supabase Auth specific columns (created_at, updated_at, etc.).RLS: Supabase automatically handles RLS for this table based on the authenticated user.2.2. user_profiles (Optional but Recommended)Purpose: To store additional user-specific profile information beyond what Supabase Auth provides, and user preferences.Columns:user_id (UUID, Primary Key, Foreign Key to auth.users.id ON DELETE CASCADE): Links to the Supabase auth user.full_name (TEXT, Nullable): User's full name.timezone (TEXT, Nullable, e.g., "Europe/Moscow"): User's preferred timezone for displaying data.preferred_units (JSONB, Nullable): e.g., {"distance": "km", "weight": "kg", "temperature": "celsius"}.created_at (TIMESTAMPTZ, Default: now()): Timestamp of profile creation.updated_at (TIMESTAMPTZ, Default: now()): Timestamp of last profile update.RLS: Enable RLS. Users can only select/update/delete their own profile.-- Example RLS Policies for user_profiles
CREATE POLICY "Users can view their own profile."
ON public.user_profiles FOR SELECT
USING (auth.uid() = user_id);

CREATE POLICY "Users can update their own profile."
ON public.user_profiles FOR UPDATE
USING (auth.uid() = user_id)
WITH CHECK (auth.uid() = user_id);
2.3. metric_sourcesPurpose: To track the origin of health data (e.g., "Android Health Connect Sync", "Oura Ring API v2", "Web Manual Entry", "Garmin Connect API").Columns:id (BIGSERIAL, Primary Key): Auto-incrementing unique identifier for the source.user_id (UUID, Not Null, Foreign Key to auth.users.id ON DELETE CASCADE): The user who owns this data source configuration.source_identifier (TEXT, Not Null): A unique machine-readable identifier for the source type (e.g., "health_connect_android", "oura_api_v2", "pihs_web_manual_supplement").source_name (TEXT, Not Null): A user-friendly name for the source (e.g., "My Pixel Phone Health Data", "Oura Ring", "Manual Supplement Log").source_type (TEXT, Not Null, e.g., "mobile_sync", "api_pull", "manual_web", "file_import"): Category of the source.configuration (JSONB, Nullable): Plugin-specific settings, e.g., API keys (encrypted if sensitive and not managed by Supabase Vault), sync frequency, last successful sync details for API sources.is_active (BOOLEAN, Default: true): Whether this source is currently active for data collection.last_synced_at (TIMESTAMPTZ, Nullable): Timestamp of the last successful data sync for this source.created_at (TIMESTAMPTZ, Default: now()).updated_at (TIMESTAMPTZ, Default: now()).Indexes: (user_id), (user_id, source_identifier) UNIQUE.RLS: Enable RLS. Users can manage their own metric sources.2.4. data_pointsPurpose: The primary table for storing all time-series health and wellness data.Columns:id (BIGSERIAL, Primary Key): Auto-incrementing unique identifier for the data point.user_id (UUID, Not Null, Foreign Key to auth.users.id ON DELETE CASCADE): The user who owns this data point.metric_source_id (BIGINT, Not Null, Foreign Key to metric_sources.id ON DELETE CASCADE): Links to the source of this data.timestamp (TIMESTAMPTZ, Not Null): The precise date and time (with timezone) when the measurement was taken or the event occurred. Crucial for time-series analysis.metric_name (TEXT, Not Null): Standardized name of the metric (see Section 3 for conventions, e.g., "deep_sleep_duration_minutes", "heart_rate_variability_rmssd_ms").value_numeric (DOUBLE PRECISION, Nullable): For numerical data.value_text (TEXT, Nullable): For categorical or simple textual data.value_json (JSONB, Nullable): For complex, nested, or multi-part data (e.g., sleep stage arrays, detailed workout structures, EEG band power sets).unit (TEXT, Nullable): The unit of measurement for value_numeric (e.g., "bpm", "minutes", "mg", "%").tags (JSONB, Nullable): An array of simple string tags for additional context or categorization (e.g., ["manual_entry", "fasted_state"]). Stored as ["tag1", "tag2"].original_payload (JSONB, Nullable): Optional: Store the raw payload from the source API or Health Connect for auditing or reprocessing.created_at (TIMESTAMPTZ, Default: now()): When the record was inserted into PIHS.Indexes:(user_id, timestamp) (Crucial for time-based queries per user)(user_id, metric_name, timestamp) (Very important for querying specific metrics over time)(metric_source_id)GIN index on tags (e.g., CREATE INDEX idx_datapoints_tags ON data_points USING GIN (tags);) for efficient tag-based queries.GIN index on value_json if specific paths within the JSON are frequently queried.Constraints: UNIQUE (user_id, metric_source_id, timestamp, metric_name) to prevent exact duplicate entries.RLS: Enable RLS. Users can manage their own data points.2.5. eventsPurpose: To log discrete occurrences, interventions, or significant life events that might influence health data (e.g., "Started new medication", "Felt ill", "Vacation start").Columns:id (BIGSERIAL, Primary Key).user_id (UUID, Not Null, Foreign Key to auth.users.id ON DELETE CASCADE).event_name (TEXT, Not Null): User-defined or standardized name of the event.start_timestamp (TIMESTAMPTZ, Not Null): When the event started.end_timestamp (TIMESTAMPTZ, Nullable): When the event ended (if applicable).duration_minutes (INTEGER, Nullable): Calculated or logged duration of the event.description (TEXT, Nullable): Detailed notes about the event.category (TEXT, Nullable, e.g., "Health", "Lifestyle", "Medication", "Diet", "Symptom").tags (JSONB, Nullable): Array of string tags.properties (JSONB, Nullable): Additional structured details specific to the event type.created_at (TIMESTAMPTZ, Default: now()).Indexes: (user_id, start_timestamp), (user_id, event_name).RLS: Enable RLS. Users can manage their own events.2.6. supplement_definitions (Optional, for Standardization)Purpose: To maintain a standardized list of supplements, their common forms, units, and categories. This helps in consistent logging and analysis.Columns:id (BIGSERIAL, Primary Key).user_id (UUID, Nullable, Foreign Key to auth.users.id ON DELETE CASCADE): If null, it's a global definition. If user-specific, it's a custom supplement.name (TEXT, Not Null, e.g., "Ashwagandha Root Extract", "Vitamin D3", "Magnesium Glycinate").common_forms (JSONB, Nullable): Array of common forms, e.g., ["capsule", "powder", "tincture"].default_unit (TEXT, Nullable, e.g., "mg", "IU", "g").category (TEXT, Nullable, e.g., "Adaptogen", "Vitamin", "Mineral", "Nootropic").description (TEXT, Nullable).created_at (TIMESTAMPTZ, Default: now()).Indexes: (name), (user_id, name) UNIQUE (a user can't have two custom supplements with the same name).RLS: Global definitions (user_id IS NULL) are readable by all. Users can manage their own custom definitions.2.7. supplement_logs (Alternative to storing all in data_points)Purpose: A dedicated table for logging supplement intake if more structured supplement-specific fields are desired beyond what data_points offers. Could link to supplement_definitions.Columns:id (BIGSERIAL, Primary Key).user_id (UUID, Not Null, Foreign Key to auth.users.id ON DELETE CASCADE).metric_source_id (BIGINT, Not Null, Foreign Key to metric_sources.id ON DELETE CASCADE).timestamp (TIMESTAMPTZ, Not Null): Time of intake.supplement_definition_id (BIGINT, Nullable, Foreign Key to supplement_definitions.id): Links to a standardized supplement.custom_supplement_name (TEXT, Nullable): If not using supplement_definition_id.dosage (DOUBLE PRECISION, Not Null).unit (TEXT, Not Null).form (TEXT, Nullable, e.g., "capsule", "sublingual").brand (TEXT, Nullable).notes (TEXT, Nullable).created_at (TIMESTAMPTZ, Default: now()).Indexes: (user_id, timestamp), (user_id, supplement_definition_id).RLS: Enable RLS.Note: Using this table adds complexity but provides more structure for supplement-specific queries. Simpler approach is to use data_points with specific metric_names and value_json for details. For initial simplicity, it's recommended to use data_points as described in Section 3.3. Data Logging Conventions (into data_points table)This section details how specific health data categories should be mapped to the data_points table. The user_id and metric_source_id will be populated based on the authenticated user and the source of the data (Android app, web manual entry).General Rule for metric_name: category_specificmeasurement_detail_unit (e.g., sleep_duration_deep_minutes, heartrate_resting_bpm). Units in metric_name are for clarity; the unit column stores the canonical unit.3.1. Sleep:Summary Metrics (one row per metric per sleep session, timestamp is typically sleep end time):metric_name: "sleep_duration_total_minutes", value_numeric: {total minutes}, unit: "minutes"metric_name: "sleep_duration_deep_minutes", value_numeric: {deep minutes}, unit: "minutes"metric_name: "sleep_duration_light_minutes", value_numeric: {light minutes}, unit: "minutes"metric_name: "sleep_duration_rem_minutes", value_numeric: {REM minutes}, unit: "minutes"metric_name: "sleep_duration_awake_minutes", value_numeric: {awake minutes during sleep period}, unit: "minutes"metric_name: "sleep_score_device", value_numeric: {score}, unit: "score" (or "0-100")metric_name: "sleep_efficiency_percentage", value_numeric: {percentage}, unit: "%"metric_name: "sleep_latency_onset_minutes", value_numeric: {minutes to fall asleep}, unit: "minutes"metric_name: "sleep_awakenings_count", value_numeric: {count}, unit: "count"Detailed Stages (one row for the entire sleep session's stage data):metric_name: "sleep_stages_array_json"timestamp: Sleep end time.value_json: [{"stage": "deep", "startTimeISO": "...", "endTimeISO": "...", "durationSeconds": ...}, ...]3.2. Heart Rate (HR) & Heart Rate Variability (HRV):metric_name: "heartrate_resting_bpm", value_numeric: {bpm}, unit: "bpm", timestamp: (usually morning)metric_name: "heartrate_average_bpm", value_numeric: {bpm}, unit: "bpm", timestamp: (can be for a day, workout, or sleep)metric_name: "heartrate_timeseries_bpm", value_numeric: {bpm}, unit: "bpm", timestamp: (for individual readings throughout the day)metric_name: "hrv_rmssd_milliseconds", value_numeric: {ms}, unit: "ms", timestamp: (often tied to sleep or a specific measurement time)metric_name: "hrv_sdnn_milliseconds", value_numeric: {ms}, unit: "ms"metric_name: "hrv_score_device", value_numeric: {score}, unit: "score"3.3. Trainings / Physical Activity:Daily Summaries:metric_name: "activity_steps_total_count", value_numeric: {steps}, unit: "steps", timestamp: (end of day)metric_name: "activity_distance_total_km", value_numeric: {km}, unit: "km"metric_name: "activity_calories_active_kcal", value_numeric: {kcal}, unit: "kcal"metric_name: "activity_minutes_moderate_total", value_numeric: {minutes}, unit: "minutes"metric_name: "activity_minutes_vigorous_total", value_numeric: {minutes}, unit: "minutes"Specific Workouts (one row per metric for a workout, timestamp is workout end time):metric_name: "workout_duration_seconds", value_numeric: {seconds}, unit: "seconds"metric_name: "workout_type_text", value_text: {"running" | "cycling" | "strength_training" | ...}metric_name: "workout_calories_total_kcal", value_numeric: {kcal}, unit: "kcal"metric_name: "workout_distance_meters", value_numeric: {meters}, unit: "meters"metric_name: "workout_heartrate_average_bpm", value_numeric: {bpm}, unit: "bpm"metric_name: "workout_details_json", value_json: {"activityType": "strength_training", "exercises": [{"name": "Squat", "sets": [{"reps": 10, "weightKg": 80}, ...]}, ...]}3.4. Stress (General - non-EEG):metric_name: "stress_level_manual_scale10", value_numeric: {1-10}, unit: "1-10_scale"metric_name: "stress_score_wearable_index", value_numeric: {score}, unit: "index"metric_name: "stress_session_duration_seconds", value_numeric: {seconds}, unit: "seconds" (if a device logs discrete stress sessions)3.5. Brain Activity (EEG, Cognitive, Emotions):EEG Band Power:metric_name: "eeg_power_alpha_abs_uv2hz", value_numeric: {power}, unit: "µV²/Hz"metric_name: "eeg_power_beta_rel_percentage", value_numeric: {power}, unit: "%" (similarly for theta, delta, gamma; abs/rel)Derived EEG Indices:metric_name: "eeg_index_focus_scale100", value_numeric: {0-100}, unit: "0-100_scale"metric_name: "eeg_index_relaxation_scale100", value_numeric: {0-100}, unit: "0-100_scale"metric_name: "eeg_index_stress_scale100", value_numeric: {0-100}, unit: "0-100_scale"metric_name: "eeg_index_concentration_scale100", value_numeric: {0-100}, unit: "0-100_scale"metric_name: "eeg_index_alphagravity_custom", value_numeric: {value}, unit: "custom_index"Cognitive Scores:metric_name: "cognitive_score_task_general", value_numeric: {score}, unit: "score"metric_name: "cognitive_score_fatigue_scale10", value_numeric: {1-10}, unit: "1-10_scale"metric_name: "cognitive_score_reverse_fatigue_scale10", value_numeric: {1-10}, unit: "1-10_scale"metric_name: "cognitive_score_selfcontrol_manual_scale10", value_numeric: {1-10}, unit: "1-10_scale"Emotions Log:metric_name: "emotion_primary_log_text", value_text: {"anger" | "joy" | "sadness" | ...}metric_name: "emotion_intensity_anger_scale10", value_numeric: {1-10}, unit: "1-10_scale" (separate entry for each emotion's intensity if logged that way)metric_name: "emotion_set_details_json", value_json: {"primary": "joy", "secondary": ["contentment"], "intensity_joy": 8, "trigger": "good_news"}3.6. Supplements:Primary Method (using data_points):metric_name: "supplement_intake_ashwagandha_mg", value_numeric: 300, unit: "mg", tags: ["oral", "ksm-66"]metric_name: "supplement_intake_vitamind3_iu", value_numeric: 5000, unit: "IU"metric_name: "supplement_intake_magnesium_glycinate_mg", value_numeric: 200, unit: "mg"For complex/custom stacks or detailed notes per intake:metric_name: "supplement_stack_morning_details_json"value_json: {"name": "Morning Focus Stack", "compounds": [{"name": "L-Theanine", "dosage": 200, "unit": "mg"}, {"name": "Caffeine", "dosage": 100, "unit": "mg"}], "method_of_intake": "oral_capsule", "notes": "Taken on empty stomach"}tags: ["nootropic", "focus"]The events table can log "Started Supplement X" or "Changed Dosage of Y".3.7. SpO2 (Blood Oxygen Saturation):metric_name: "spo2_average_percentage", value_numeric: {98.5}, unit: "%"metric_name: "spo2_minimum_percentage", value_numeric: {95.0}, unit: "%"metric_name: "spo2_reading_percentage", value_numeric: {99.0}, unit: "%" (for individual spot checks)3.8. Weather:metric_name: "weather_temperature_ambient_celsius", value_numeric: {22.5}, unit: "°C"metric_name: "weather_humidity_ambient_percentage", value_numeric: {60}, unit: "%"metric_name: "weather_condition_description_text", value_text: "partly_cloudy"metric_name: "weather_airpressure_ambient_hpa", value_numeric: {1012}, unit: "hPa"This detailed schema and logging convention will provide a solid foundation for data storage and subsequent analysis in your PIHS platform using Supabase.
